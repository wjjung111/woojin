# CLAUDE.md — ARAP 프로젝트 지침서

> Claude Code가 이 폴더에서 작업할 때 **자동으로 먼저 읽는 파일**.
> 모든 작업은 이 문서의 원칙을 따른 뒤에 시작한다.

---

## 1. 목적

감정평가사가 의견서 1건 작성에 드는 시간을 **1시간 이상 → 30분 이내**로 줄인다.
대상 영역은 **집합건물 의견서 → 괄호감정표/명세표 → 토지건물 ARAP** 순서로 확장.

## 2. 이 앱의 성격 (가장 중요한 설계 원칙)

**이 앱은 "AI가 판단해주는 도구"가 아니다. "평가사 판단 외 모든 단순작업을 자동화하는 도구"다.**

### 자동화 대상 (앱이 처리)
- 시점수정 계산 (지수 조회·기간·산식)
- 가치형성요인 비교표 생성
- 표준 문구 삽입 (반복되는 정형 문장)
- 산식·반올림·유효숫자 처리
- DOCX 템플릿 채우기·서식 적용·다운로드

### 자동화 비대상 (평가사 수동 유지) — **절대 자동화 시도 금지**
- 비교사례 선정 (어떤 거래사례를 쓸지)
- 개별요인 우열 판단 (외부·내부·호별·기타 비교치 결정)
- 최종 시산가액 결정 (조정 여부·결정단가)
- 필요도 논리 문구 (조건부 의견, 특수사정 기술 등 케이스별 서술)

> 새 기능 제안 시 이 분류부터 확인할 것.
> "AI가 비교사례 자동 추천" → 목표 밖.
> "평가사가 비교사례 고른 뒤 자동으로 시점수정·요인표·문구·DOCX까지 한 방에" → 목표 안.

## 2-1. 타겟 사용자

**현재**: 본인 (개발자 = 사용자 본인, 1인 사무용).
**향후 가능성**: 외부 배포 시 UI·온보딩·안정성 보강 필요. 지금은 본인 워크플로우 기준으로 만든다.

## 2-2. 우선순위 (2026-06-30 기준)

1. ~~[단기] 비주거용/집합건물 의견서~~ → **완료** (단일/여러호수 × 주거/비주거 + 혼합 + 시점수정 지수 자동계산까지, v1.2)
2. [중기] 괄호감정표 + 명세표 일괄 자동화 (여러호수 모드엔 아직 미구현 — 샘플 받으면)
3. [장기] 토지건물 ARAP (지가지수/지가변동률도 R-ONE API에 있음 — [[project_rone-index-api]])

> 새 작업 들어올 때 위 우선순위와 부합하는지 먼저 확인.
> 주거용 의견서는 **현재 만족스러운 상태** — 안정성 유지 위주, 큰 변경 자제.

## 3. 작업 원칙 (필수)

### 3-1. 운영 중인 파일은 절대 직접 건드리지 않는다
- `ARAP_v1.0_주거용.html` 같은 **현재 사용 중인 버전은 직접 수정 금지**.
- 패치는 반드시 **버전을 올려서 사본**으로 작업한다 (`v1.0` → `v1-1` → `v1-2` ...).
- 수정 전에는 항상 **백업 사본**을 만든다 (`*.bak_YYYYMMDD_HHMMSS`).

### 3-2. 변경 후 반드시 사용자 검증을 거친다
- 코드만 수정해놓고 "끝났습니다"라고 말하지 않는다.
- 사용자가 **브라우저 새로고침(Ctrl+Shift+R) → 실제 다운로드 → 결과 확인**까지 한 다음에야 작업 종료.
- DOCX 생성 로직은 단순 코드 리뷰로 검증 불가 → **반드시 실제 파일 열어서 페이지 수·순서·내용 확인**.

### 3-3. 사용자는 코딩을 잘 모른다
- 변경 내용은 **한국말로, 결과 위주로** 설명한다.
- "정규식이 어쩌고" 같은 기술 용어는 **왜 문제였는지 한 줄**만 덧붙이고, 사용자가 할 일(새로고침, 재업로드 등)을 **숫자로 명시**한다.
- 모르는 게 있으면 추측하지 말고 묻는다.

### 3-4. 위험한 작업은 사전 확인
- 전면 리팩토링, 대규모 코드 이동, localStorage 키 변경, 템플릿 교체 등은 **시작 전에 사용자 동의** 필요.
- 1500줄 짜리 단일 HTML 앱은 작은 실수도 전체가 멈춘다 — **작게, 자주, 검증하며** 바꾼다.

## 4. 코드 스타일

### 4-1. 현재 스타일 = "압축형 단일 HTML"
- 빌드 도구 없음. React + Babel + lodash 전부 CDN. **브라우저에서 그대로 실행**되는 단일 HTML.
- 한 줄에 if/return/할당 다 붙여쓰는 압축 스타일. 의도된 선택이므로 **존중**한다.
- 새 코드 작성 시 기존 스타일을 따라간다(주변 코드 보고 맞춤). 본인이 짠 코드 한가운데 갑자기 정석 스타일을 박지 않는다.

### 4-2. 단, 다음은 예외 (안전 우선)
- **빈 `catch(e){}` 금지** — 최소한 `catch(e){console.error('맥락설명',e);}` 형태.
- **정규식으로 XML 자르는 코드**(`<w:p>` 같은 거) 추가/수정 시 — `</w:p>` 등 닫는 태그를 넘지 못하게 가드 패턴(`(?:(?!<\/w:p>)[\s\S])*?`) 사용. 오늘 5페이지 사고가 정확히 이거.
- **새 함수/변수명**은 약어로 시작하지 않는다. 기존에 `pNum`, `fU`, `uD` 같이 있어도 새로 만들 땐 `parseNumber`, `finalUnitPrice` 식으로. 약어와 풀이름 공존 OK.

### 4-3. 주석
- WHAT(이 코드가 뭐하는지)은 안 쓴다. 함수명이 말해줘야 함.
- WHY(왜 이렇게 짰는지, 특히 비직관적 선택)만 한 줄로 남긴다.
- 예: `// {{사례선정이유}}는 fixReasonParagraph가 멀티런 서식까지 처리 — fixPlaceholders보다 먼저 실행`

## 5. 위험 지점 (Landmines)

작업하기 전에 반드시 인지할 것.

| 위치 | 문제 | 대응 |
|---|---|---|
| `fixReasonParagraph` (1089행 부근) | 정규식이 `</w:p>` 경계를 안 막아서 본문 통째로 삼킨 적 있음 | `(?:(?!<\/w:p>)[\s\S])*?` 패턴 유지 |
| `createZipStored` (804행) | CRC32 생략 — unzip이 경고 띄움 | Word는 무시하지만 향후 까다로운 docx 리더에서 깨질 수 있음 |
| 빈 `catch(e){}` 12군데 | 에러 삼킴 — 디버깅 불가 | 수정할 때마다 console.error 추가 |
| `TPL_KEY` localStorage 캐시 | 망가진 템플릿이 캐시되면 새 코드 배포해도 사용자에겐 옛날 게 로딩됨 | 임베드 템플릿 교체 시 **반드시 키 버전도 올림** (`v3` → `v4`) |
| `Date.now()+Math.random()*9999+i` ID | 충돌 가능 | 새 코드에선 `crypto.randomUUID()` 권장 |
| `DEFAULT_TPL` (770행, 72KB 한 줄) | base64로 docx 통째 임베드 — 라인 길이 73KB | 절대 들여쓰기 도구 돌리지 말 것. 통째 한 줄 유지. |
| 명세표 xlsx `calcChain.xml` | 셀 덮어쓴 뒤 계산체인이 남으면 엑셀이 '복구' 경고. Word보다 엄격 | `downloadMyeongseXlsx`에서 calcChain.xml+Content_Types override+rel을 **항상 제거**(엑셀이 재생성). deltaMS 조건에 걸지 말 것 |
| 명세표 구조/층별면적 | 반드시 **등기** 표제부(1동의 건물의 표시) 기준. 건축물대장 아님 | 구조=`ov.regStructure`(등기), 층별면적=`ov.floorDetail`(등기). PDF프롬프트도 등기 전용 명시됨 |

## 6. 테스트 방법 (DOCX 다운로드 변경 시)

1. 브라우저에서 HTML 열기
2. 강력새로고침: **Ctrl+Shift+R** (localStorage 캐시까지 새로 로드)
3. 4단계까지 진행 → Word 다운로드
4. **반드시 열어서 확인**:
   - 페이지 수 (정상은 보통 8~15페이지 정도)
   - **1페이지가 "본건 건물의 개요"로 시작**하는지 (← 5페이지 사고 재발 방지)
   - 플레이스홀더 `{{...}}`가 남아있지 않은지
   - 사례선정이유 문단의 밑줄/굵게 서식이 살아있는지

## 7. 폴더 구조 / 주요 파일

> **루트에는 앱 실행 필수 4종만** 둔다. `ARAP 실행.bat`→`arap_launch.ps1`→`arap_집합건물v1.2.html`+`arap_index_data.js`가
> **같은 폴더(루트)에 붙어있어야** 실행됨(html이 `arap_index_data.js`를 상대경로로 로드). 이 4개는 **하위폴더로 옮기지 말 것.**
> 템플릿 .docx는 html에 base64 내장이라 런타임에 안 불러옴 → `템플릿/`으로 옮겨도 앱 정상.

```
woojin/ (저장소 루트)
├── arap_집합건물v1.2.html              ← ★현재 운영본 (직접수정 금지 → v1-3로 사본). 루트 고정
├── ARAP 실행.bat                       ← ★사용자 진입점: 지수 자동수신(arap_launch.ps1) 후 v1.2 열기. 루트 고정
├── arap_launch.ps1                    ← 부동산원 R-ONE API에서 지수 수신 → arap_index_data.js 생성 (키 내장). 루트 고정
├── arap_index_data.js                 ← 매매가격지수 데이터 (자동생성 — 직접수정 금지). 루트 고정
├── CLAUDE.md                          ← 이 파일
├── 템플릿/                            ← 현역 의견서 템플릿 4종(.docx) + 표지·청구서·명세표 시제품(.xlsx)
│                                        (주거+비주거+여러호수 혼합본은 html에 base64 임베드)
├── 문서/                              ← 작업중_여러호수N호_설계.md (N호확장·시점수정 작업 인수인계 상세)
├── 아카이브/                          ← 구버전 html(v1.1)·백업(.bak_*)·개발잔재(launch.json) 등
└── 토지건물/                          ← 향후작업(토지건물 ARAP)용 자료: 토지건물_산출근거.xlsx
```

> **현재 상태(2026-06-30)**: 집합건물 의견서 = 단일/여러호수 × 주거/비주거 + **주거·비주거 혼합**(호별 용도, 시점수정 사례별 자본수익률/매매지수)까지 완성. 엔진 핵심: `expandHoRows`(호 N개 표 복제), `expandSijeom`(사례별 시점수정 블록 복제), `normalizeMarkers`(쪼개진 마커 복원).
>
> **(2026-07-12) v1.2 운영본 = 시점수정 지수 자동계산 + 명세표 등기기준 고정**: 부동산원 R-ONE Open API로 아파트(`A_2024_00045`, 서울 구단위)·연립다세대(`A_2024_00080`, 권역단위만 공표 → 구 선택 시 자동 권역 대체) 매매가격지수를 받아, 시점수정치 모달의 [지수 자동계산] 패널이 산식 텍스트를 자동 생성. **규칙(확정): 거래시점·기준시점 둘 다 전월 지수 적용, 시점수정치 여섯째자리 반올림(부동산원 KAS 조회화면과 동일), 미공표월은 최근 공표분 대체, 날짜는 한글표기(점표기 YYYY.MM은 파서 오인 유발이라 금지).** 브라우저에서 API 직접 호출은 CORS 차단 → `ARAP 실행.bat`이 앱 열기 전에 `arap_launch.ps1`로 지수를 수신(`arap_index_data.js`)하는 구조. 데이터 파일 없어도 앱은 정상 동작(자동계산만 안내문구). 갱신: 파일 1일 초과 시 백그라운드 재수신. **명세표 층별면적(floorDetail)은 PDF 인식 프롬프트에서 등기 표제부 전용으로 고정**(건축물대장 층별현황 사용 금지 — 값이 달라 명세표 틀어짐). 상세는 [[작업중_여러호수N호_설계.md]] §14.

## 8. 자주 하는 실수 (방지 체크리스트)

- [ ] 운영본(현재 v1.2)에 직접 수정했는가? → **STOP, v1-3로 복제부터**
- [ ] 백업 만들었는가? → **STOP, 백업부터**
- [ ] localStorage `TPL_KEY` 안 올리고 임베드 템플릿만 바꿨는가? → 사용자는 새 템플릿 못 봄
- [ ] 정규식에 `[\s\S]*?` 썼는데 경계 가드 빠졌는가? → 5페이지 사고 재현
- [ ] 빈 `catch(e){}` 새로 추가했는가? → console.error 한 줄 추가
- [ ] 다운로드 받아서 실제로 열어봤는가? → **안 열어봤으면 작업 미완**

## 9. 모르는 건 묻는다

이 문서에 없는 상황(새로운 기능 추가, 구조 변경, 외부 라이브러리 도입 등)은
**추측해서 진행하지 말고 사용자에게 묻는다.** 사용자는 코딩을 잘 모르므로,
물을 때는 **결과 위주로 2~3개 선택지**를 제시한다.
