## 1. 뭐가 이미 완료됐나 (다시 만들지 말 것!)

### 앱 (exact_집합건물v1.0.html) — ✅ 완료 + 헤드리스 크롬 E2E 검증까지 끝남
- 양 IIFE에 헬퍼 추가: `capData / capTypeList / capRegionList / capLatestDisp / buildNrCapText` (각 IIFE의 `buildSjIdxText` 바로 아래)
- `buildNrCapText(유형,지역,거래일,기준일)` → 자본수익률 복리 산식 텍스트 생성:
  - 첫 분기 = 거래일~분기말 일수비례(당일포함, 예 `56/91`), 끝 분기 = 분기초~기준일 일수비례(당일포함), 중간 = `(1+r)` 복리
  - **미고시(미공표) 분기 = 최근 공표분 갈음** + 분기 목록에 `(YYYY년 Q분기 자료)` 표기
  - 시점수정치 = 여섯째자리 반올림 → 다섯째 표시. 기존 예시 `≒1.05087` 정확 재현 확인함
  - 생성 텍스트는 `parseNrCapDetail` 완전 호환 (다호수 사례별 시점수정 경로 그대로 탐)
- UI: ③사례입력 모달·④SijeomModal(단일+다호수 4곳)의 [지수 자동계산] 패널이 `propType==="비주거용"`이면 **[자본수익률 자동계산]**(유형·지역 select + 계산)으로 전환. 단일 ④ 노란카드엔 필드 직접 채우는 패널 추가(nrType/nrRegion/nrMigosi/nrMigosiPrev/nrPeriod/nrCapPct/nrCapVal/nrCapFormula)
- localStorage: `arap-nrcap-kind / arap-nrcap-region` (유형·지역 기억)
- E2E 검증 완료: 사례 붙여넣기→선정→기준시점→비주거 전환→계산→소수 1.05609·산식·미고시 채움, 콘솔에러 0

### 데이터 계약 (앱 ↔ ps1)
```
window.ARAP_INDEX_DATA.capReturn = {
  fetchedAt, source,
  types: { "집합상가": { latest: "20261",              // YYYYQ
                        regions: { "서울": { "20242": 0.56, ... } } },  // 값 = %(퍼센트), 앱이 /100
           "오피스": …, "중대형상가": …, "소규모상가": … } }
```

### arap_launch.ps1 — 코드 완료, **실데이터 검증만 남음**
- `Invoke-Rone`: 60초 타임아웃 + 3회 재시도 (주거·비주거 공용 — 기존 30초 타임아웃이 전체 수신을 죽이던 문제 해결)
- `Fetch-CapReturn`: 통계목록에서 `임대동향`+`수익률` 이름 매칭 → 유형별(집합/중대형/소규모/오피스, 오피스텔 제외) 모든 연도구간 표 병합 → **DTACYCLE_CD=QY 필수**(안 그러면 응답 null!) → **페이지네이션(1000행/p, 최대 40p)** → 행 필터: `ITM_NM`에 '자본수익률'만, `CLS_FULLNM` 깊이 ≤2(시도 단위만) → `regions[지역][YYYYQ] = %값(소수2)`
- `Update-CapReturnOnly` + `-CapOnly` 스위치: 주거 매매지수 안 받고 자본수익률만 병합 (테스트용)
- 워크플로(update-index.yml)에 `cap_only` 체크박스 입력 추가됨

## 2. 실측으로 이미 확인된 사실 (Actions 로그에서)

- 자본수익률 표는 **유형 × 연도구간**으로 분할: 구세대 `A_2024_00394~00417` + `00686/687`(2002~2012), **신세대(2024Q3~ 개편) = "임대동향 수익률(분기)(2024년3분기~)_*" (T2… 코드)** — 신세대는 이름에 '자본수익률'이 없어서 필터를 넓힌 것
- 행 구조: `WRTTIME_IDTFR_ID=202201`(=2022 1분기, YYYY0Q), `CLS_NM=전체/서울/부산…`, `ITM_NM=자본수익률`, `DTA_VAL=1.03`(=1.03%, **퍼센트 스케일 확정**)
- 페이지네이션 전 실측: 4유형 모두 수신됐으나 최신 분기가 2022~2023에서 잘림(1000행 제한) → 페이지네이션 코드는 넣었지만 **아직 실측 못 함** ← 이게 남은 전부

## 3. 남은 일 (순서대로)

1. **로컬에서 데이터 수신 실측**: 저장소 폴더에서
   ```
   powershell -ExecutionPolicy Bypass -File arap_launch.ps1 -CapOnly
   ```
   (키는 `arap_apikey.local.txt` 필요 — 로컬 개발폴더에 이미 있음. 클론 폴더면 복사)
2. 콘솔 확인 포인트:
   - 후보 표 ~30건+ (신세대 T코드 4개가 목록에 보여야 함)
   - `[진단]` 줄의 ITM_NM·DTA_VAL 정상 여부
   - 각 유형 `최신 202XX분기` — **2025Q4~2026Q2 수준까지 올라와야 성공**. 2022~2023에 머물면 페이지네이션/신세대표 파싱 문제 → 신세대 표 응답 구조(분기 표기가 다를 수 있음: `2024Q3` 등 — `ConvertTo-QKey`가 웬만한 표기는 흡수함) 확인해서 조정
   - **분기 연속성**: 서울 기준 2013~최신까지 구멍 없는지 (구세대 2024Q1·Q2 + 신세대 2024Q3~ 이음새 주의)
3. `arap_index_data.js`의 `capReturn` 값을 부동산원 공표치와 대조 (집합상가 서울 분기 대략 ±0.n% 수준)
4. **앱 검증**: Ctrl+Shift+R → ④ 물건유형 비주거용 → 노란카드 [자본수익률 자동계산] 계산 → 필드 채움 확인. ③/④ 시점수정 모달의 계산 버튼도. **시점수정치를 손계산(또는 기존 수동산식)과 대조**
5. 실제 케이스 Word 출력 열어서 자본수익률 섹션 확인 (CLAUDE.md §6 체크리스트)
6. 통과하면: 검증 결과를 설계문서 §15에 한 줄 추가 → push → **PR #35** 리뷰·머지 (머지하면 Pages 반영)
