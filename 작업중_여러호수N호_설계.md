## 4. 설계 — "엔진이 가호행을 틀로 N복제"
템플릿 새로 안 만들고 **기존 COM의 가호행을 틀로 재사용**. 엔진이:
1. buildData가 호별 데이터 배열 `d.__hoData={개요:[{...}],가치:[...],적용단가:[...],시산:[...],결정:[...]}` 생성. 각 항목 = 그 호의 (가호 마커 기준) 값 묶음.
2. `expandHoRows(xml, hoData)` 신규: 각 표의 가호행(틀)을 찾아 → hoData 배열 길이만큼 복제, 복제본마다 가호마커(Y_013 등)를 그 호 값으로 직접 치환 → 나/다/라행 삭제 → 틀자리에 복제본 삽입. 가치형성은 2행 묶음.
3. generateDocxBlob에서 trimHoRows 대신 expandHoRows 호출 (fixPlaceholders 앞).
4. 합계행: buildData가 _ucs 전체 합산(이미 _totalAmt 호수무관) → C_041/C_056 그대로.

장점: 템플릿 리스크 최소(기존 검증본), 호 무제한.
주의(랜드마인): XML 행복제 정규식은 `</w:tr>` 경계 가드 필수. 마커 치환은 복제 후 즉시(중복마커 방지).

- **진단(preview 2002, #3·#4 선정)**: 임시디버그로 buildData의 `적용_사례번호="#3, #4"` 정확 확인(window.__bd), `_symStr`(3926)도 동일. 그런데 최종 docx는 "#3"만. 파이프라인 추적(__tr)에서 fixReasonParagraph가 '거래사례 기호'를 안 만듦(afterReason=NO).
- **근본원인**: 사용자 템플릿의 사례선정 문장은 `{{사례선정이유}}`가 아니라 **`{{Y_233}}~{{Y_238}}` 개별 마커**로 구성(python으로 DEFAULT_TPL_COM 확인). 그래서 fixReasonParagraph({{사례선정이유}})는 no-op, fixPlaceholders가 Y_233~238을 채움. 그런데 buildData의 Y_237 채우기(옛 4154)가 `sym=_refC?.symbol||_shownC[0]?.symbol` **대표 1개만** 넣고 있었음.
- **수정(4153행 부근)**: `d["Y_236"]=""`(# 은 적용_사례번호에 포함), `d["Y_237"]=d["적용_사례번호"]||(sym?…)`, `d["Y_238"]=d["적용_조사"]||…`. 이미 만든 적용_사례번호("#3, #4")·적용_조사 재사용. → 문장 "거래사례 기호#3, #4을 …".
- **검증**: 코드 논리(Y_237=적용_사례번호=검증된 "#3, #4"). 임시디버그(__bd, __tr) 전부 제거. 실물검증은 사용자 11호 재출력에 위임.
- ⚠️ **다음 세션 경고**: preview 검증 시 root-multi의 'Word 다운로드' 버튼을 반복 클릭하면 **OS "다른 이름으로 저장" 창이 매번 뜸**(사용자 극혐). export 검증은 값 확인으로만 하거나, 반드시 `HTMLAnchorElement.prototype.click`(download속성 분기) + `window.showSaveFilePicker`를 몽키패치로 막아 저장을 차단하고 `__lastBlob`만 캡처할 것. 실제 다운로드 트리거 금지.
